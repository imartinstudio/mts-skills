# 深度代码审查

以审查者身份工作：只读、不给未经证实的问题定罪、不直接修改代码，除非用户另行要求。

## 范围与准备

1. 确认审查对象：默认审查当前工作区的 staged 与 unstaged diff；用户指定基线、提交或 PR 时以其为准。
2. 读取仓库级和相关包的 `AGENTS.md`、代码地图或贡献指引。先用 `git diff --stat`、`git diff --name-only` 和 `rg -n` 确定最小阅读范围。
3. 阅读每个改动 hunk 的完整函数/类型边界，并只在必要时向上一层调用方、数据契约、迁移逻辑和相邻测试扩展。
4. 保持只读：不修改工作树、不创建提交、不发送外部请求。不要输出密钥、令牌、cookie 或真实用户数据。

## 审查清单

逐项寻找可复现、可定位的问题，而不只检查语法：

- **行为与兼容性**：旧输入、旧文件布局、缓存、CLI/API 响应、失败语义、幂等性和部分完成后的状态是否仍兼容。
- **隐藏副作用**：额外网络/LLM 调用、费用、写文件、日志泄露、缓存失效、状态写入顺序、不可逆删除和并发覆盖。
- **边界情况**：空值、空数组、缺失文件、畸形 JSON、重复项、超长内容、Unicode/路径、重试、取消及异常分支。
- **安全与性能**：路径穿越、注入/XSS、认证边界、敏感信息外泄、无界读取/日志/循环、同步 I/O、重复昂贵调用。
- **可维护性**：命名和注释是否准确描述真实语义；是否存在重复分支、隐式协议、弱类型断言或难以测试的内联逻辑。
- **测试**：新增或改变的成功、失败、边界及兼容路径是否覆盖；测试是否只证明最顺利的路径。

将“改动前已存在”的问题与本次 diff 引入或扩大影响的问题区分开。对于推断，明确写为推断并说明依据。

## 验证

1. 先运行仓库文档指定的最小相关测试和类型检查；若类型检查失败，记录阻断错误，不要假定后续测试已覆盖运行时行为。
2. 必要时增加只读验证，例如 `git diff --check`、目标测试、构建或静态搜索。不要为了 review 启动浏览器或访问生产环境，除非用户明确要求。
3. 如果没有发现问题，说明已检查的范围、未运行的验证以及残余风险；不要用“看起来没问题”代替证据。

## 输出格式

按严重程度从高到低输出：`P0`（阻断构建/数据或安全事故）、`P1`（高概率功能破坏或严重副作用）、`P2`（应在合并前修复的稳定性/兼容性问题）、`P3`（维护性或低风险问题）。

每条问题必须包含：

- 可点击的文件与单一行号；
- 简洁标题；
- 触发条件、实际后果和为何由本次改动引入或放大；
- 必要时给出最小修复方向，而不粘贴大段重写代码。

最后单列“测试与覆盖”：已执行命令及结果、未覆盖的关键路径。没有问题时，明确写 `未发现阻断性问题`，再列残余风险。
